## 摘要
设计有效的 Lean 证明智能体是形式化数学推理的核心挑战。除了构建更强的证明器，近期工作也关注围绕 Lean 的工作流：智能体如何分解证明义务、使用工具和编译器反馈、诊断失败、修复证明以及维护结构化的证明上下文。受代码层自进化智能体启发，我们研究这样的工作流是否能够通过进化而非手工设计来生成。我们提出了一种自进化的 Lean 证明智能体，其中一个小型的、固定的、可信的运行时包裹着一个完全可变的工件区：包含证明工作流、提示和工具。与大多数针对固定外部基准进行优化的自进化系统不同，我们的系统使智能体与其基准协同进化。在世代之间，得分最高的智能体（冠军）通过一个节制掌握的课程更新机制来修正活跃的任务分布，该机制仅在当前难度被掌握后才引入更难的证明义务；同时通过单锚点重校准，在更新后的基准上重新运行冠军，以在难度上升时保持得分的可比性。所有进化都保持在一个 Lean 验证循环内：无论智能体如何重写自身，只有当其行为在可信快照下产生通过 Lean 验证的证明时才算成功，且每次尝试必须输出一个机器可读的、基于 Lean 的证明上下文，其表示形式可以进化，但其基础性得到强制执行。我们运行了协同进化轨迹和固定基准的基线各 15 个活跃世代，并在保留的 miniF2F 测试集上进行对比。最佳的协同进化智能体达到了 45.1% 的保留测试解决率，而初始种子为 12.7%，最佳固定基准智能体为 32.0%，表明在协同进化基准下，基于验证器的自进化能够改进 Lean 证明工作流。

## Abstract
Designing effective Lean proof agents is a central challenge in formal mathematical reasoning. Beyond building stronger provers, recent work emphasizes the workflow around Lean: how an agent decomposes proof obligations, uses tools and compiler feedback, diagnoses failures, repairs proofs, and maintains structured proof context. Motivated by code-level self-evolving agents, we study whether such workflows can be evolved rather than hand-designed. We present a self-evolving Lean proof agent in which a small fixed, trusted runtime wraps a fully mutable workspace: the proof workflow, prompts, and tools. Unlike most self-evolving systems, which optimize against a fixed external benchmark, our system coevolves the agent and its benchmark. Between generations, the highest-scoring agent (the champion) revises the active task distribution through a mastery-throttled curriculum update that introduces harder proof obligations only after the current level is mastered, and a single-anchor recalibration re-runs the champion on the updated benchmark to keep scores comparable as difficulty rises. All evolution stays inside a Lean-grounded verification loop: however the agent rewrites itself, a success counts only when its behavior yields Lean-verified proofs under a trusted snapshot, and each attempt must emit a machine-readable, Lean-grounded proof context whose representation may evolve but whose groundedness is enforced. We run the coevolving trajectory and a fixed-benchmark baseline for 15 active generations and compare them on a held-out miniF2F test split. The best coevolving agent reaches a 45.1% held-out solve rate, versus 12.7% for the seed and 32.0% for the best fixed-benchmark agent, showing that verifier-grounded self-evolution can improve Lean proof workflows under a coevolving benchmark.

## 2. 论文提出的方法论

### 2.1 可变工作区与可信运行时分离

- 系统分为两部分：
  - **固定的可信运行时**：负责 Lean 验证、评估、基准管理、证明上下文校验等决定正确性和奖励的关键机制，不参与进化搜索。
  - **完全可变的工作区**：包含证明工作流（如修复循环、分解策略）、工具函数、提示模板等，是智能体自进化的对象，通过受限的编辑接口进行修改。
- 代码有效性与证明成功解耦：变异后的智能体首先通过代码执行与冒烟测试，然后才进入评估阶段；定理是否被证明完全由可信运行时重新使用受信任的 Lean 快照进行验证。

### 2.2 证明上下文合约：表示自由，篡改困难

- 每次证明尝试必须输出三元组 `{ok, proof_body, proof_context}`。
- **Proof_context** 记录证明的数学结构（如断言、引理、子目标、依赖关系），其表示形式（图、树、表格等）可以自由进化，但必须满足一组最低限的**根基化要求**：根节点存在、至少包含一个数学声明、依赖边指向已知项、求解项必须有 Lean 验证证据等。
- **防欺骗机制**：可信运行时重新验证最终证明和证明上下文证书，忽略所有可变代码的自报告；禁止在证明体中顶层定义无关定理；对 Lean 命令进行净化，拒绝 `sorry` 和 `admit`；保护不可变文件，检测篡改则回滚；可变入口点在隔离的工作进程上运行。

### 2.3 基准协同进化

- 基准由候选池按难度分层（L1: 单策略热身引理，L2: miniF2F 验证集，L3: PutnamBench）抽取得到，初始混合比使种子智能体解决率约为 37%。
- 每代结束后，仅当**冠军智能体**的原始解决率超过阈值 $\gamma = 0.30$ 时才更新基准：
  - **退休规则**：只退休所有智能体都解决了的最多 $\kappa$ 个任务。
  - **替换规则**：若冠军在退休任务所在的难度层级 $\ell$ 的解决率 $m_\ell \ge \tau = 0.70$，则用下一层级 $\ell+1$ 的新任务替换；否则从同层级横向替换。这形成**掌握节流**效应，防止基准突然变难。
  - 替换时优先选择历史上失败次数最多的领域中的任务，反映群体弱点。
- **单锚点重校准**：基准变难后，将冠军在新基准上重评估，按公式 $c_{t+1} = c_t \cdot r_t / \max(r_{t+1}, \epsilon)$ 更新难度系数，所有智能体的归一化得分 $q = r \times c$，使跨代得分可比。

### 2.4 父代选择

- 采用基于 DGM 的加权采样，权重 $w = \max(q, 0) \cdot \frac{1}{1 + n_c}$，其中 $q$ 为难度归一化得分，$n_c$ 为已产生（含待评估）的子代数量，以此奖励高能力且避免少数谱系过度繁殖。
- 采用**流式归档**：每个被接受的子代完成后立即归档，并可作为同一代后续子代的父代。

## 7. 优点

- **新颖的协同进化范式**：将自进化主体与基准共同进化，通过掌握节流和单锚点重校准避免训练信号消失和分数不可比问题，对形式化证明领域具有开创性。
- **验证器接地**：自始至终将 Lean 作为不可篡改的信任锚点，防止智能体在自修改中“作弊”，保证所有声称的证明都经过可信验证，安全性强。
- **细致的演化谱系分析**：不仅报告最终得分，还深入分析了工作流、工具、节点验证和证明上下文在代际间的变化，展示了许多有趣的自发设计（如 binder 隔离、自包含子目标验证）及其淘汰原因，为理解自进化动态提供了丰富材料。
- **可扩展的架构**：可信运行时与可变工作区的分离清晰，有利于未来扩展到其他形式验证环境或引入更丰富的奖励信号。

## 8. 不足与局限

- **单次运行缺乏统计显著性**：所有定量结论均基于一次协同进化轨迹，没有多次重复实验或置信区间，结论的稳健性有待验证。
- **与最强手工设计系统的差距依然巨大**：最佳进化智能体在 miniF2F-test 上仅 45.1%，而如 Goedel-Architect 等手工设计系统可达 99.2%，说明纯自进化尚未触及上限。
- **稀疏奖励导致工作流浅层化**：进化压力几乎完全落在最终证明验证上，导致智能体偏好短平快的修复循环，反而抑制了更深层的证明分解与组装工作流的发展；如果不对证明结构引入额外奖励，可能难以涌现高级证明规划能力。
- **基准演化能力有限**：目前基准的演化仍由固定的规则（冠军阈值、掌握率）和手工预分层的问题池驱动，并未让智能体自主生成或提出新问题；长期来看，更理想的协同进化应包含问题生成能力。
- **硬件与资源限制**：受限于远端 API 调用和单机架构，种群大小与世代数受限（仅 15 代，每代至多 3 个被接受的子代），可能限制了探索广度与深度。
