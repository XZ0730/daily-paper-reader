## 摘要
生产环境中的LLM智能体常常在每个请求上重复生成相同程序步骤的代码，浪费延迟和可靠性。我们用一条智能体工具制造流水线取代了这种推理时的编码循环，该流水线在部署前将重复的标准操作程序步骤编译成经过验证的、带版本的工具。工具制造者扎根于在线环境，收集执行轨迹、观察后端模式和取值、生成候选工具，并根据标注案例进行修复。在运行时，生产智能体直接调用这些工具，仅在需要时才回退到代码生成。我们将该方法部署在一个配送中心告警分诊系统中，智能体在此根据跨异构指标后端的44节点标准操作程序诊断告警。在生产环境中，工具调用将p50延迟降低了42%。在1500个历史告警上，通过抑制重复步骤中的逐次运行差异，端到端错误率最多降低了53%。由于工具返回紧凑的结构化判定结果，它们还促成了更简单的直接调用架构，在一项对照剥离实验中进一步将p50延迟降低了62%。带版本的工具也改善了可审计性，并暴露出规范缺口与上游数据漂移。我们的结果表明，能够自我进化的智能体可以使工业LLM系统更快、更可靠、更易于运维。

## Abstract
Production LLM agents often waste latency and reliability by regenerating code for the same procedural steps on every request. We replace this inference-time coding loop with an agentic tool-making pipeline that compiles repeated SOP steps into validated, versioned tools before deployment. The tool-maker grounds synthesis in the live environment as it collects execution traces, observes backend schemas and values, generates candidate tools, and repairs them against labeled cases. At runtime, the production agent calls these tools directly and falls back to code generation only when needed. We deploy the approach in a Fulfillment Center alarm-triage system, where an agent diagnoses alarms against a 44-node SOP over heterogeneous metric backends. In production, tool calls reduce p50 latency by 42%. On 1,500 historical alarms, they reduce end-to-end error rate by up to 53% by suppressing run-to-run variance in repeated steps. Because tools return compact structured verdicts, they also enable a simpler direct-call architecture, reducing p50 latency by a further 62% in a controlled ablation. Versioned tools also improve auditability and expose specification gaps and upstream data drift. Our results show that self-evolving agents can make industrial LLM systems faster, more reliable, and easier to operate.

### 1. 论文的核心问题与整体含义

*   **核心问题**：在生产环境中，基于代码生成的LLM智能体（CodeAct范式）在每次处理请求时，都需要在推理阶段重新编写和执行代码。当面对稳定、重复的业务流程（如标准操作程序，SOP）时，这种重复解释指令、重新发现数据模式（Schema）并生成类似代码的方式，导致了严重的**延迟、高昂的计算成本和不可靠性（每次运行结果不一致）**。
*   **整体含义**：该研究旨在将LLM智能体从“每次都需要重新思考如何做”的模式，转变为“将重复经验编译为可复用的固化技能”的**自我进化**模式。通过在部署前将标准操作步骤编译成预先验证的工具，可以**摊销成本**，将推理时的重复性工作降至最低，从而让工业级LLM系统在速度、可靠性和可维护性上实现质的飞跃。

### 2. 论文提出的方法论

*   **核心思想**：提出了一套**智能体工具制造流水线**。该流水线将一个分层的多智能体架构中，推理时由子智能体反复生成和运行代码的循环，替换为**构建时一次性生成、测试、修复**的确定性和版本化工具。运行时，主智能体直接调用这些工具，仅在工具缺失或失败时才回退到代码生成模式。
*   **关键技术细节与流程**：
    1.  **构建时工具合成**：
        *   **数据采集子智能体**：针对SOP中的每个决策节点，在真实生产环境（通过MCP）中运行，采样少量标注案例并收集**执行轨迹**。轨迹包含成功与失败案例的查询代码、MCP返回的完整数据模式（字段名、类型、值范围）以及最终判定结果。
        *   **工具制造者LLM**：基于**SOP节点文本、其在决策树中的位置、以及数据采集器提供的执行轨迹**，生成候选工具代码。
        *   **反思-修复循环**：将生成的候选工具在完整的标注训练集上进行测试。对于失败的案例，由**反思者LLM**生成故障诊断，然后驱动**工具制造者LLM**根据诊断和失败案例进行重写。此循环最多迭代三轮，最终部署在训练集上通过率最高的版本。
    2.  **运行时架构**：
        *   **工具调用模式**：主智能体在遍历决策树时，直接调用对应节点的工具，工具返回结构化的判定结果（如`true`、`false`、`no data`）、观测值、阈值和文本解释。
        *   **直接调用架构**：由于工具输出极其紧凑（少量Token），论文进一步去除了中间的**子智能体层**，由主智能体直接调用工具，从而避免了主-子智能体间的通信开销。
    3.  **智能体微调策略**：
        *   为确保小模型智能体在工具不可用时的鲁棒性，采用混合轨迹的教师蒸馏：教师模型在75%的轨迹中使用工具（其中25%的工具随机停用），在25%的轨迹中完全不使用工具。只保留那些观测、根因和行动决策均正确的轨迹，用于微调学生模型。

### 6. 论文的主要结论与发现

*   **显著的延迟降低**：在生产环境中，用工具调用替换代码生成循环，使p50延迟**降低42%**；若配合简化的直接调用架构，p50延迟总计可**降低88%**（相对于原始基线）。
*   **可靠性的提升**：端到端错误率相对降低**36%~53%**，主要归因于确定性工具消除了代码生成中的`run-to-run`差异（如使用了错误指标或错误报告结论）。
*   **流水线关键组件不可或缺**：数据采集轨迹和反思修复循环对达到高通过率是**必需而非可选的**。脱离真实环境数据的反思修复甚至可能损害性能。
*   **瓶颈转移至规范质量**：当工具生成接近完美时，**剩余的瓶颈从模型能力转移到了SOP文本本身的规范缺口和模糊性上**。补充训练数据和澄清SOP能将Pass@1从94.5%提升至99.9%。
*   **运维优势**：带版本的工具通过记录确定性的输入/输出，意外地**暴露了上游数据和环境的不一致性**（如数据类型错误、字段含义变化），实现了被动态代码生成所掩盖的**数据漂移监控**。

### 7. 优点

*   **实际问题驱动**：紧密围绕工业界真实的延迟和可靠性痛点，提供了端到端的部署方案。
*   **方法论设计精巧**：
    *   **“两步走”的自我进化模式**（离线编译，在线调用）清晰且高效，完美平衡了实时性的严苛要求和代码智能体的灵活性。
    *   **扎根环境的合成与修复**是关键亮点，通过执行轨迹而非仅文本描述来生成工具，极大弥合了从文本到可执行代码的鸿沟。
    *   **反思-修复循环**的设计确保了生成工具的质量，并且该机制在获取外部数据反馈后才能发挥最大效用。
*   **额外收益**：工具带来的审计性、一致性以及对上游数据漂移的被动发现能力，是超出性能指标本身的、极具价值的“意外收获”。

### 8. 不足与局限

*   **应用范围局限于单一场景**：方法只在“发运码头告警分诊”这一个任务上进行了验证，其在其他类型的结构化工作流（如自由形式的运维手册）中的泛化能力尚未证实。
*   **人工在环的依赖**：尽管系统实现了自动化工具生成和漂移检测，但其应用仍依赖**人工审核**以确保生产安全，并且**澄清模糊的SOP文本**本质上是一个需要人类意图输入的任务，难以完全自动化。
*   **工具生成的监督需求**：目前生成工具仍需要**带标注的训练案例**。虽然论文尝试了无标注方案并达到了接近水位的性能，但要完全去除对人工标注的依赖仍有距离。
